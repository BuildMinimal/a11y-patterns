# Pattern: Icon-Only Button — Color Contrast

**Category:** Color Contrast
**WCAG:** 1.4.11 Non-Text Contrast — Level AA

---

## What this pattern demonstrates

How to ensure icon-only buttons meet WCAG contrast requirements — covering the icon color against its background, hover states, and proper accessible names for screen reader users.

---

## Why it matters

Icon-only buttons are ubiquitous in modern UI: close modals, delete items, toggle menus, share content, and more. However, they present two distinct accessibility challenges:

1. **Visual contrast:** The icon must be clearly visible against its button background. A 3:1 contrast ratio is required for non-text content (WCAG 1.4.11). Many designs use subtle icon colors that fail this requirement, making buttons invisible to users with low vision or color vision deficiency.

2. **Accessible naming:** Screen reader users cannot "see" icons. An icon-only button without an accessible name is announced simply as "button" — completely useless. The user has no idea what the button does.

The failure mode is concrete: a user with low vision sees a button but cannot tell what icon is on it, or worse, cannot see the button at all. A screen reader user hears "button" and has no context to decide whether to activate it.

---

## WCAG Criterion

**1.4.11 Non-Text Contrast — Level AA**

> The visual presentation of the following have a contrast ratio of at least 3:1 against adjacent colors:
>
> - User Interface Components: Visual information required to identify user interface components and states, except for inactive components or where the appearance of the component is determined by the user agent and not modified by the author;
> - Graphical Objects: Parts of graphics required to understand the content, except when a particular presentation of graphics is essential to the information being conveyed.

**4.1.2 Name, Role, Value — Level A**

> For all user interface components (including but not limited to: form elements, links and components generated by scripts), the name and role can be programmatically determined; states, properties, and values that can be set by the user can be programmatically set; and notification of changes to these items is available to user agents, including assistive technologies.

Source: [Understanding Success Criterion 1.4.11](https://www.w3.org/WAI/WCAG21/Understanding/non-text-contrast.html)

The `before/` version shows a toolbar with icon-only buttons that have two problems:

1. **Insufficient contrast:** Icon colors are too light against their backgrounds, failing the 3:1 requirement
2. **No accessible names:** Buttons lack `aria-label` or `title` attributes, so screen readers announce them as "button" with no context

**Specific failures in `before/styles.css`:**

| `.btn--delete` icon   | `#ef4444`  | `#fee2e2`  | 1.88:1   | FAIL             |
| `.btn--share` icon    | `#9ca3af`  | `#f3f4f6`  | 1.23:1   | FAIL             |
| `.btn--favorite` icon | `#9ca3af`  | `#f3f4f6`  | 1.23:1   | FAIL             |

**HTML accessibility failures:**

- No `aria-label` attributes on any buttons
- No `title` attributes as fallback
- Screen readers announce: "button, button, button, button" — completely useless

## Key rules

**1. Icon-only buttons must have accessible names.**
Use `aria-label` to provide a clear, concise description of what the button does. The label should describe the _action_, not the icon. Use "Close dialog" not "X icon" or "Close button".

**2. Icons must meet 3:1 contrast against their background.**
This is a lower threshold than text (4.5:1) because icons are simpler shapes, but many designs still fail it. Test your icon colors with a contrast checker.

**3. Hover and focus states must also meet contrast requirements.**
Don't fix the default state but leave hover/focus with insufficient contrast. Every state a user can experience must be accessible.

**4. Prefer text labels when space permits.**
Icon-only buttons should be a last resort. If you have space, add a visible text label alongside the icon. This benefits all users, not just those using assistive technology.

## Design tokens

| Token                            | Value     | Usage                                   | Contrast            |
| -------------------------------- | --------- | --------------------------------------- | ------------------- |
| `color.icon.neutral`             | `#4b5563` | Default icon color on light backgrounds | 4.62:1 on `#f3f4f6` |
| `color.icon.danger`              | `#dc2626` | Delete/remove icon color                | 3.56:1 on `#fee2e2` |
| `color.button.background`        | `#f3f4f6` | Neutral button background               | —                   |
| `color.button.danger-background` | `#fee2e2` | Danger button background                | —                   |
| `color.button.hover-background`  | `#e5e7eb` | Neutral button hover state              | —                   |

## Testing

```bash
# Run the axe tests for this pattern
npm test -- patterns/01-color-contrast/icon-only-button

# Validate token contrast ratios
npm run validate-tokens
```

Expected test output:

```
✓ before/ has color-contrast violations
✓ after/ has no color-contrast violations
```

You can also verify the contrast ratios manually:

- [WebAIM Contrast Checker](https://webaim.org/resources/contrastchecker/)
- [Colour Contrast Analyser](https://www.tpgi.com/color-contrast-checker/) (desktop app, supports eyedropper)

**Manual screen reader testing:**

1. Open `after/index.html` in a browser
2. Use NVDA (Windows) or VoiceOver (Mac)
3. Tab through the icon buttons
4. Verify each button is announced with its action: "Close dialog, button", "Delete item, button", etc.
